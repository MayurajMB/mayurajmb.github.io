📖 1. Introduction & Context

From Basic FHIR APIs to Secure, Interoperable Health Apps — Why SMART on FHIR Exists and Why It Matters for Every Healthcare Developer.

1. The Historical Problem: A Decade of "Custom Integrations"

Before healthcare data standardization, every hospital had its own proprietary way of exposing patient data. If you wanted to build a third-party clinical decision support app, you would negotiate a unique data feed with Epic, then a completely different one with Cerner, then yet another with Allscripts. You were not building software — you were building an integration business with each hospital as its own full-time project.

Year Standard / Event What it Solved What it Left Unsolved
1987 HL7 v2 Basic lab result messages between hospital departments. Verbose, binary, difficult to parse. No REST API.
2011 HL7 v3 / CDA Structured XML clinical documents (C-CDA). Extremely complex XML schemas. Still no REST API.
2014 FHIR R2 Released Modern JSON REST API + standardized resource types (Patient, Observation, etc.). No defined security/authentication model. Left to implementers.
2016 SMART on FHIR v1 Defined OAuth 2.0 + OIDC as the universal security layer on top of FHIR. Scope granularity was limited. v1 scopes like patient/*.read were coarse.
2020 21st Century Cures Act (US) Mandated SMART on FHIR compliance for all certified US EHRs. Made it law. —
2022 SMART on FHIR v2 Granular CRUD scopes (patient/Observation.rs), asymmetric JWT auth, stronger security. —

2. The Security Gap in Base FHIR

FHIR explicitly specifies that it does not define security protocols. It is built expecting a security layer to be applied on top of it. Without SMART, any organization implementing FHIR would have to answer these questions independently:

Base FHIR Provides

  • Standard JSON resource schemas (Patient, Obs).
  • RESTful GET, POST, PATCH endpoints.
  • Search parameters and query syntax.

Base FHIR Does NOT Provide

  • Authentication ("Who are you?" — Identity).
  • Authorization ("What can you see?" — Permissions).
  • Context (Which patient/visit is active?).

SMART on FHIR Adds

  • OpenID Connect for Identity proof.
  • OAuth 2.0 scopes for granular Permissions.
  • Launch parameters for active EHR Context.

3. The Regulatory Mandate

21st Century Cures Act (ONC Final Rule, 2020): This US federal law mandated that all ONC-certified EHR systems (i.e., all major hospital EHRs) must expose SMART on FHIR R4 APIs by 2022. Non-compliance results in "information blocking" violations. This is why Epic, Cerner, and Allscripts must implement what we're covering today.

🧠 2. What is SMART on FHIR?

Substitutable Medical Applications, Reusable Technologies — coined by Harvard's Dr. Ken Mandl in 2010.

1. The USB Analogy: Type-C & Display over Type-C

The clearest way to understand the relationship between FHIR and SMART on FHIR is through the USB Type-C standard.

FHIR = USB Type-C

USB Type-C defines the physical connector standard — the shape of the port, the pin layout, and the basic power/data protocol. Every device that has a Type-C port can at minimum charge and transfer data.

Similarly, FHIR defines the data connector standard — the shape of clinical data (Patient, Observation), the REST API contract, and the query language. Any EHR with a FHIR endpoint exposes data in a predictable way.

  • Universal physical connector shape.
  • Basic power + data protocol included.
  • Does NOT define what flows over the wire (video? audio? identity?).

SMART on FHIR = Display over Type-C

DisplayPort Alt Mode over Type-C is an additional protocol that rides on top of the same USB-C connector — but now the port can carry a full 4K video signal. The connector didn't change. The cable didn't change. A capability layer was added on top.

Similarly, SMART on FHIR rides on top of FHIR — it adds the security and identity capability layer (OAuth 2.0, OIDC, Scopes) on top of the same FHIR endpoints, without changing the data model at all.

  • Same FHIR port — no change to data model.
  • Adds authentication + authorization on top.
  • Unlocks rich, secure data flows (like Display unlocks 4K).

2. The Web Analogy: HTTP & HTTPS

For developers, the most familiar analogy is the evolution from HTTP to HTTPS on the web.

FHIR = HTTP

HTTP defines how web browsers and servers communicate — the request/response format, status codes, headers, and URL structure. It is purely about content and transport, not about security.

Similarly, FHIR defines the clinical data request/response format — GET /Patient/{id}, JSON resources, search parameters. It's the data protocol, not the security layer.

  • Defines the data format & endpoint structure.
  • Works out of the box — just make a REST call.
  • No encryption, no identity, no access control.

SMART on FHIR = HTTPS

HTTPS is HTTP with TLS encryption layered on top. The web page content (HTML) didn't change. The URL structure didn't change. But now every request is encrypted, authenticated, and tamper-proof.

Similarly, SMART on FHIR is FHIR with OAuth 2.0 + OIDC layered on top. The FHIR resources didn't change. The endpoints didn't change. But now every request is authorized, the user is authenticated, and data is protected.

  • Same FHIR endpoints — zero data model changes.
  • Adds encryption + identity proof on top.
  • Without SMART, FHIR is like HTTP — open to abuse.

3. The EHR Vendor Ecosystem

Each major EHR vendor has their own branded portal and implementation, but they all follow the same SMART on FHIR specification under the hood. This is the key insight.

3. The Trinity of Standards — Combined

A core strength of SMART is that it doesn't invent novel security. It assembles three proven, massive internet standards into a cohesive profile. Each layer has a precisely defined role and hands off to the next.

FHIR (HL7 R4)

The Language of Clinical Data

  • Defines JSON schemas (Patient, Obs).
  • Provides RESTful search parameters.
  • Defines the server URLs your app calls.

OAuth 2.0 (RFC 6749)

The Authorization Layer

  • Handles asking user for explicit consent.
  • Safely issues temporary Access Tokens.
  • Manages granular scope-based permissions.

OpenID Connect (OIDC)

The Identity Layer

  • Proves who the logged-in user is.
  • Issues ID Tokens (always a JWT).
  • Provides the fhirUser claim pointing to their FHIR profile.

⚙️ 3. Core Standards Deep Dive

Understanding the mechanics of OAuth 2.0, OpenID Connect (OIDC), and JSON Web Tokens (JWTs) in the context of healthcare.

1. The Foundation: OAuth 2.0 Actors

OAuth 2.0 is an authorization framework that enables applications to obtain limited access to user accounts on an HTTP service. To understand how a SMART app gets access to an EHR, you must first understand the four key participants in any OAuth transaction.

Resource Owner

Who: The Patient or the Clinician.

Role: They are the entity capable of granting access to a protected resource. When a patient logs into a portal to approve an app, they are acting as the Resource Owner.

Client

Who: Your SMART Application.

Role: The application making protected resource requests on behalf of the resource owner and with its authorization. Can be a mobile app, web SPA, or a backend service.

Authorization Server

Who: The EHR's OAuth/Login Portal (e.g., Epic MyChart, Cerner Ignite).

Role: The server issuing access tokens to the client after successfully authenticating the resource owner and obtaining authorization.

Resource Server

Who: The EHR's FHIR API.

Role: The server hosting the protected resources (clinical data), capable of accepting and responding to protected resource requests using access tokens.

2. The Identity Layer: OpenID Connect (OIDC)

OAuth 2.0 was designed exclusively for authorization (granting access to APIs), not authentication (proving exactly who the user is). Because platforms needed to know user identity, they started writing custom, non-standard hacks on top of OAuth. This led to a fragmented ecosystem.

OpenID Connect (OIDC) is a standardized identity layer built directly on top of OAuth 2.0. By adding OIDC to the SMART on FHIR flow, apps can securely authenticate the user and authorize data access in a single unified step, receiving two distinct types of tokens.

Token Type Protocol Primary Purpose Analogy Format
Access Token OAuth 2.0 Used in the Authorization: Bearer header to fetch FHIR data from the Resource Server. Instructs the API what the app is allowed to do. A Valet Key. It lets the valet drive your car, but doesn't tell them who you are or let them open the glovebox. Often an Opaque String (meaningless to the client app).
ID Token OpenID Connect Provides cryptographic proof to the Client application of exactly who just logged in, when they logged in, and how. A Driver's License. It unequivocally proves your identity to the bouncer at the door. Always a JSON Web Token (JWT).

3. Unpacking the ID Token: JSON Web Tokens (JWT)

OIDC strictly specifies that the ID Token must be formatted as a JSON Web Token (JWT). A JWT is a compact, URL-safe means of representing claims to be transferred between two parties. It consists of three base64-url encoded parts separated by periods (.).

Header.Payload.Signature

eyJhbGciOiJSUzI1NiIsImtpZCI6IjFhMmIzYzRkIn0.eyJpc3MiOiJodHRwczovL2VociIsImV4cCI6MTY5OX0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

1. The Header (JOSE Header)

Tells the receiving server how to verify this token — which algorithm was used to sign it, and which specific public key to look up from the JWKS endpoint.

{
  "x5t#S256": "A3kPz-XKstu9w7B5iF8vxn58WiPqZGtu8EJH4VboZit",
  "kid":      "jwt",
  "alg":      "RS256"
}
x5t#S256 SHA-256 thumbprint of the signing certificate. Used to locate the exact key.
kid Key ID. Points to the matching public key in the EHR's JWKS endpoint.
alg Signing algorithm. RS256 = RSA with SHA-256 (asymmetric). Never HS256 in production SMART.

2. The Payload (Claims)

Contains all identity and authorization information issued by ECW after a successful SMART EHR launch. This is the decoded payload of a real id_token.

{
  "sub":         "c5b11811-45b8-xxxx-xxxx-00505684ce1b_68976711-xxxx",
  "iss":         "https://staging-fhir.ecwcloud.com/fhir/r4/FFBJCD",
  "aud":         ["jlIyWPTcKz175xxxxxxxxxxxxxxxxx", "https://staging-fhir.ecwcloud.com/fhir/r4/FFBJCD"],
  "fhirUser":    "https://staging-fhir.ecwcloud.com/fhir/r4/FFBJCD/Practitioner/xxxx",
  "clientName":  "MAIA",
  "workflow":    "launch",
  "vendorId":    "2961",
  "scope":       ["patient/Patient.read", "patient/Condition.read", "openid", "launch", "..." ],
  "iat":         1773042664,
  "exp":         1773042964,
  "nbf":         1773042664,
  "jti":         "a0807002-xxxx-4960-xxxx-11a37a19a9c9"
}
sub Unique stable ID of the logged-in user (Practitioner). Scoped per EHR + Practice.
iss Issuer — the full FHIR base URL of the ECW tenant. Must match your discovery URL.
aud Audience — your client ID AND the FHIR server URL. Both must be validated by your app.
fhirUser SMART-specific. Absolute URL to the Practitioner FHIR resource. Fetch this to get demographics.
workflow ECW-specific. Tells you the launch mode — launch = EHR Embed Launch.
scope The exact SMART scopes that were granted. Always validate this matches what your app requested.
iat / exp / nbf Unix timestamps — issued at, expires at, not valid before. Token is valid for 5 minutes only.

3. The Signature

To create the signature part, you take the encoded header, the encoded payload, a secret, the algorithm specified in the header, and sign that.

For example, if using HMAC SHA256:

HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret
)

This guarantees the token hasn't been altered by a Man-in-the-Middle.

4. Deep Dive: SMART on FHIR Payload Claims

When an EHR issues an ID token during a SMART launch, the payload contains specific standard OIDC claims alongside a very critical SMART-specific claim.

Example Decoded SMART ID Token Payload
{ "iss": "https://ehr.hospital.org/oauth2", "sub": "Practitioner/12345", "aud": "my-app-client-id", "iat": 1700000000, "exp": 1700003600, "fhirUser": "https://ehr.hospital.org/fhir/Practitioner/12345" }

Security Claims

  • iss (Issuer): The URL of the EHR Authorization Server that issued the token. Must exactly match the EHR's known endpoint.
  • aud (Audience): Your app's client_id. If this doesn't match your app, the token was stolen from another app and must be rejected.

Lifecycle Claims

  • iat (Issued At): Unix timestamp of when the token was created.
  • exp (Expiration): Unix timestamp of when the token dies. Your backend must reject the token if the current time is past this value.

Identity Claims

  • sub (Subject): A unique, stable identifier for the user within the EHR (often their internal DB ID).
  • fhirUser: (SMART Specific) The absolute FHIR URL to read the user's demographic profile! Example: .../fhir/Patient/789.

🔍 4. SMART Configuration & Discovery

How applications dynamically discover an EHR's specific security endpoints without hardcoding URLs.

1. The Auto-Discovery Mechanism

Unlike standard APIs where endpoints (like /oauth/token) might be hardcoded in application logic, SMART on FHIR relies on dynamic discovery. Because every hospital might configure their EHR slightly differently, SMART requires the EHR to host a JSON configuration file at a standard location relative to the FHIR Base URL.

The Discovery URL: [FHIR_BASE]/.well-known/smart-configuration

2. Anatomy of the Discovery JSON

When an app boots up, it fetches this JSON file to learn exactly what capabilities the EHR server supports, where security endpoints live, and what authentication methods are required.

Example SMART Config Response
{ "authorization_endpoint": "https://fhir.epic.com/interconnect-fhir-oauth/oauth2/authorize", "token_endpoint": "https://fhir.epic.com/interconnect-fhir-oauth/oauth2/token", "token_endpoint_auth_methods_supported": [ "client_secret_basic", "private_key_jwt" ], "capabilities": [ "launch-ehr", "launch-standalone", "client-public", "client-confidential-asymmetric" ], "scopes_supported": ["patient/*.read", "offline_access", "openid", "fhirUser"] }

Endpoints

Provides the exact URLs your app must redirect users to (authorization_endpoint) and where it must POST to get tokens (token_endpoint).

Auth Methods

Tells your app how it must authenticate itself. E.g., private_key_jwt means the EHR requires asymmetric RS384 signatures for backend launches.

Capabilities

An array of feature flags. If an EHR supports launch-ehr, it means it can embed your app natively in the clinician workspace.

3. Client Registration Paradigms

Before any app can utilize the discovered endpoints, the EHR must know it exists. The EHR issues a unique client_id during Registration.

Registration Type Process Common Use Case
Static Registration Developer manually logs into the EHR Vendor's Developer Portal (e.g., Epic App Orchard, Cerner Code), fills out a form, uploads JWKS URLs, and is issued a static client_id. 99% of production healthcare applications.
Dynamic Registration The app programmatically sends an HTTP POST request to the EHR's registration endpoint with its metadata and is instantly issued a client_id over the wire. Automated test suites, sandbox environments. Rarely supported in live production EHRs.

📱 6. Standalone Launch

For patient-facing mobile apps and Web SPAs launched outside the EHR network.

1. The Use Case

A patient downloads a diet tracker on their iPhone. They click "Connect to My Hospital". The app starts "cold" and doesn't know who the patient is yet. It must guide them to the hospital's login page securely.

2. Sequence Diagram: Auth Code Flow with PKCE

Because the app lives on the open public internet, it uses the Authorization Code flow enhanced with PKCE.

sequenceDiagram participant U as User (Patient) participant A as Mobile App (Client) participant Auth as EHR Auth Server participant FHIR as EHR FHIR API A->>A: Generate PKCE Verifier & Hash it into Challenge A->>U: Redirect to EHR Login URI U->>Auth: User sees Epic/Cerner Login Screen Note over A,Auth: Request includes client_id, scopes, and PKCE Challenge U->>Auth: User logs in & grants app consent Auth->>A: Redirect back to App with Auth Code A->>Auth: POST /token (Sends Code + Original PKCE Verifier) Auth->>Auth: Verifies Verifier matches initial Challenge Auth->>A: Returns Access Token & ID Token A->>FHIR: GET /Patient (Header: Bearer Token) FHIR->>A: Returns FHIR JSON

3. What is PKCE? (Proof Key for Code Exchange)

Because an iPhone app or a Single Page Application (SPA) cannot safely embed a hardcoded client_secret (since anyone could inspect the code and steal it), they must use PKCE to dynamically prove they are the legitimate app.

1. The Promise (Challenge)

Before redirecting the user, the app creates a highly random secret (Verifier), hashes it (Challenge), and sends the Challenge to the EHR. It's safe to send over the URL because mathematicians cannot reverse a secure hash.

2. The Interception Risk

If a malicious app on the user's phone intercepts the URI redirect containing the Auth Code, it will try to instantly trade that code for an Access Token at the /token endpoint.

3. The Proof (Verifier)

The EHR requires the app trading the code to present the original, unhashed Verifier. The malicious app doesn't know the Verifier. The legitimate app does. The EHR issues the token to the legitimate app and denies the attacker.

🏥 7. Embed Launch (EHR Launch)

For single-page applications embedded directly inside a clinician's EHR interface (often rendered within an iFrame or integrated browser window).

1. The Use Case and The "Magic" of Context

Scenario: Dr. House is currently viewing Patient John Doe's chart inside Cerner PowerChart. He navigates to a designated tab and clicks "Open Growth Chart SMART App".

Unlike a Standalone launch, the app does not show a login screen. It doesn't even ask the doctor to search for John Doe. The app instantly opens and displays John Doe's data. This magical inheritance of active context is the hallmark of an Embed Launch.

2. Sequence Diagram: The EHR Launch Flow

sequenceDiagram participant Doc as Clinician participant EHR as EHR UI (Epic/Cerner) participant A as SMART App (iFrame) participant Auth as EHR Auth Server Doc->>EHR: Opens Patient Chart & Clicks App Link EHR->>A: Navigates to App's /launch URL (Query: ?launch=abc&iss=https...) Note over A: App saves 'launch' token locally A->>Auth: Redirect to /authorize (passes client_id & launch=abc) Auth->>Auth: Reads 'launch' token to remember active Patient Note over Auth: Clinician is already logged into EHR. Auto-SSO! Auth->>A: Redirect back to App with Auth Code A->>Auth: POST /token (Sends Code) Auth->>A: Returns Access Token + Context (patient_id, encounter_id)

3. Breaking Down the Context Payload

The primary architectural difference is the launch parameter. The EHR generates this opaque string right before it opens the app's iframe. The app blindly passes this string back to the Auth Server during the OAuth handshake.

Because the Auth Server generated the launch token, it knows exactly which chart the doctor was looking at. When the final Access Token is returned, the EHR attaches the crucial contextual IDs directly to the JSON response.

Example Token Response Endpoint JSON
{ "access_token": "eyJhbGciOiJSUzI1NiIs...", "token_type": "Bearer", "expires_in": 3600, "scope": "patient/*.read openid fhirUser", "patient": "12b3c4d5", // App immediately knows the active patient! "encounter": "987654321", // App knows the active visit/encounter "smart_style_url": "https://..." }
The Golden Rule of Embed: With an Embed Launch, your application's UI should never prompt the user to search for a patient. Your app must strictly honor the patient ID passed in the token exchange response to prevent accidentally showing the wrong patient's data in the chart view.

⚙️ 8. Backend Launch (System to System)

A high-security authentication workflow designed exclusively for server daemons, CRON jobs, and bulk data extraction without human users.

1. The Problem: No Human "Consent"

In Standalone and Embed launches, a human always clicks an "I Agree" button. For background tasks (like syncing all appointments overnight), there is no human. Therefore, the server application itself must mathematically prove its identity to the EHR without relying on a shared password (client secret).

2. Sequence Diagram: Two-Legged JWT Bearer Flow

sequenceDiagram participant App as Backend App (Daemon) participant Auth as EHR Auth Server participant FHIR as EHR FHIR API Note over App: Cron Job Starts App->>App: 1. Generate local JWT payload App->>App: 2. Sign JWT using local RSA Private Key App->>Auth: 3. POST /token (Sends signed JWT + client_id) Auth->>App: 4. HTTP GET App's public JWKS URL App-->>Auth: Responds with Public Key Set Auth->>Auth: 5. Verify JWT Signature using Public Key Auth->>App: 6. Issue Access Token (No human consent needed) App->>FHIR: 7. GET /Appointment (Header: Bearer Token) FHIR->>App: Returns thousands of FHIR JSON records

3. Defining the Cryptographic Trust

The entire backend flow rests on Public Key Cryptography (Asymmetric Encryption). Here's how building that trust works:

1. The Keypair

You generate an RSA or EC Keypair. The Private Key never leaves your secure server environment. It is the absolute secret.

2. The JWKS (JSON Web Key Set)

You convert the corresponding Public Key into a standardized JSON format (JWKS) and host it securely on the internet where the EHR can reach it.

3. The Signed Assertion

Instead of sending a password to /token, you construct a JWT containing your client_id, sign it with your Private Key, and send that. The EHR fetches your Public Key to mathematically verify you are the true owner of the Private Key.

Example Payload Your Server Must Sign
{ "iss": "my-client-id", "sub": "my-client-id", "aud": "https://ehr.hospital.org/oauth2", "exp": 1700003600, "jti": "random-uuid-xyz-123" }

🛡️ 5. SMART Scopes & Permissions

Granular, predictable access control to ensure minimal required data is accessed.

1. The Evolution: v1 vs v2 Scopes

When an application requests access, it must define exactly what it needs to do. In early SMART versions (v1), scopes were broad and often misunderstood. SMART v2 introduced a much more granular, standardized format.

SMART v2 Scope Syntax Format
[context] / [resource] . [action] ? [optional parameters]

Context

Defines the bounding box of the data. patient means data belonging to a single specific patient. user means data the currently logged-in clinician can see. system means unrestricted bulk access.

Resource

The exact FHIR Resource Type (e.g., Observation, Encounter, Condition). You can also use * to request all resources.

Action

The CRUD operation allowed. Standard actions are c (create), r (read), u (update), d (delete), or s (search).

2. Clinical Scope Translators

Translating clinical needs into scopes requires understanding exactly which FHIR resource holds what data. Here is a cheat sheet of common requirements for apps:

Requested Clinical Data Required FHIR Resource(s) Example Scope Request
Patient Demographics Patient patient/Patient.rs
Vitals, Labs, Smoking Status Observation patient/Observation.rs
Allergies & Intolerances AllergyIntolerance patient/AllergyIntolerance.rs
Medications & Prescriptions MedicationRequest, MedicationStatement patient/MedicationRequest.rs
Write Notes to Chart DocumentReference patient/DocumentReference.c

3. Essential Non-Clinical Scopes

Beyond clinical data, your app must request foundational scopes to make the launch work successfully.

  • openid & fhirUser: Critical for enabling the OpenID Connect flow. Instructs the server to return an ID token with the user's demographic reference.
  • offline_access: Requests a Refresh Token. Without this, your app will lose access the moment the initial access token expires (usually after 15-60 minutes).
  • launch/patient: Instructs the EHR to trigger a "Patient Selector" UI during login so the provider can choose which patient chart they want the app to open.
