📖 1. Introduction & Context

From Basic FHIR APIs to Secure, Interoperable Health Apps — Why SMART on FHIR Exists and Why It Matters for Every Healthcare Developer.

1. The Historical Problem: A Decade of "Custom Integrations"

Before healthcare data standardization, every hospital had its own proprietary way of exposing patient data. If you wanted to build a third-party clinical decision support app, you would negotiate a unique data feed with Epic, then a completely different one with Cerner, then yet another with Allscripts. You were not building software — you were building an integration business with each hospital as its own full-time project.

Year Standard / Event What it Solved What it Left Unsolved
1987 HL7 v2 Basic lab result messages between hospital departments. Verbose, binary, difficult to parse. No REST API.
2011 HL7 v3 / CDA Structured XML clinical documents (C-CDA). Extremely complex XML schemas. Still no REST API.
2014 FHIR R2 Released Modern JSON REST API + standardized resource types (Patient, Observation, etc.). No defined security/authentication model. Left to implementers.
2016 SMART on FHIR v1 Defined OAuth 2.0 + OIDC as the universal security layer on top of FHIR. Scope granularity was limited. v1 scopes like patient/*.read were coarse.
2020 21st Century Cures Act (US) Mandated SMART on FHIR compliance for all certified US EHRs. Made it law. —
2022 SMART on FHIR v2 Granular CRUD scopes (patient/Observation.rs), asymmetric JWT auth, stronger security. —

2. The Security Gap in Base FHIR

FHIR explicitly specifies that it does not define security protocols. It is built expecting a security layer to be applied on top of it. Without SMART, any organization implementing FHIR would have to answer these questions independently:

Base FHIR Provides

  • Standard JSON resource schemas (Patient, Obs).
  • RESTful GET, POST, PATCH endpoints.
  • Search parameters and query syntax.

Base FHIR Does NOT Provide

  • Authentication ("Who are you?" — Identity).
  • Authorization ("What can you see?" — Permissions).
  • Context (Which patient/visit is active?).

SMART on FHIR Adds

  • OpenID Connect for Identity proof.
  • OAuth 2.0 scopes for granular Permissions.
  • Launch parameters for active EHR Context.

3. The Regulatory Mandate

21st Century Cures Act (ONC Final Rule, 2020): This US federal law mandated that all ONC-certified EHR systems (i.e., all major hospital EHRs) must expose SMART on FHIR R4 APIs by 2022. Non-compliance results in "information blocking" violations. This is why Epic, Cerner, and Allscripts must implement what we're covering today.

🧠 2. What is SMART on FHIR?

Substitutable Medical Applications, Reusable Technologies — coined by Harvard's Dr. Ken Mandl in 2010.

1. The USB Analogy: Type-C & Display over Type-C

The clearest way to understand the relationship between FHIR and SMART on FHIR is through the USB Type-C standard.

FHIR = USB Type-C

USB Type-C defines the physical connector standard — the shape of the port, the pin layout, and the basic power/data protocol. Every device that has a Type-C port can at minimum charge and transfer data.

Similarly, FHIR defines the data connector standard — the shape of clinical data (Patient, Observation), the REST API contract, and the query language. Any EHR with a FHIR endpoint exposes data in a predictable way.

  • Universal physical connector shape.
  • Basic power + data protocol included.
  • Does NOT define what flows over the wire (video? audio? identity?).

SMART on FHIR = Display over Type-C

DisplayPort Alt Mode over Type-C is an additional protocol that rides on top of the same USB-C connector — but now the port can carry a full 4K video signal. The connector didn't change. The cable didn't change. A capability layer was added on top.

Similarly, SMART on FHIR rides on top of FHIR — it adds the security and identity capability layer (OAuth 2.0, OIDC, Scopes) on top of the same FHIR endpoints, without changing the data model at all.

  • Same FHIR port — no change to data model.
  • Adds authentication + authorization on top.
  • Unlocks rich, secure data flows (like Display unlocks 4K).

2. The Web Analogy: HTML & HTTPS

For developers, the most familiar analogy — HTML is the data format, HTTPS is the secure protocol that protects it.

FHIR = HTML

HTML is the data format of the web. It defines the structure and meaning of content — headings, paragraphs, links, images. A browser can render HTML, a scraper can read it, but HTML itself has zero security — anyone can copy or intercept it.

Similarly, FHIR is the data format of healthcare. It defines the structure and meaning of clinical data — Patient resources, Observation JSON, FHIR search parameters. Any app can call a FHIR endpoint, but FHIR itself has no authentication or authorization built in.

  • Defines structure & meaning of data.
  • Universally readable — just make a REST call.
  • No security, no identity, no access control.

SMART on FHIR = HTTPS

HTTPS wraps HTTP + HTML in TLS encryption. The HTML content didn't change. The URL structure didn't change. But now the entire communication is encrypted, the server identity is verified, and the data stream is tamper-proof.

Similarly, SMART on FHIR wraps FHIR in OAuth 2.0 + OIDC. The FHIR resources didn't change. The endpoints didn't change. But now every request requires authorization, the user's identity is proven, and granular data access is enforced.

  • Same FHIR data format — nothing changes.
  • Adds identity + authorization on top.
  • Without SMART, FHIR is like plain HTML — unprotected.

3. The Trinity of Standards — Combined

A core strength of SMART is that it doesn't invent novel security. It assembles three proven, massive internet standards into a cohesive profile. Each layer has a precisely defined role and hands off to the next.

FHIR (HL7 R4)

The Language of Clinical Data

  • Defines JSON schemas (Patient, Obs).
  • Provides RESTful search parameters.
  • Defines the server URLs your app calls.

OAuth 2.0 (RFC 6749)

The Authorization Layer

  • Handles asking user for explicit consent.
  • Safely issues temporary Access Tokens.
  • Manages granular scope-based permissions.

OpenID Connect (OIDC)

The Identity Layer

  • Proves who the logged-in user is.
  • Issues ID Tokens (always a JWT).
  • Provides the fhirUser claim pointing to their FHIR profile.

⚙️ 3. Core Standards Deep Dive

Understanding the mechanics of OAuth 2.0, OpenID Connect (OIDC), and JSON Web Tokens (JWTs) in the context of healthcare.

1. The Foundation: OAuth 2.0 Actors

OAuth 2.0 is an authorization framework that enables applications to obtain limited access to user accounts on an HTTP service. To understand how a SMART app gets access to an EHR, you must first understand the four key participants in any OAuth transaction.

Resource Owner

Who: The Patient or the Clinician.

Role: They are the entity capable of granting access to a protected resource. When a patient logs into a portal to approve an app, they are acting as the Resource Owner.

Client

Who: Your SMART Application.

Role: The application making protected resource requests on behalf of the resource owner and with its authorization. Can be a mobile app, web SPA, or a backend service.

Authorization Server

Who: The EHR's OAuth/Login Portal (e.g., Epic MyChart, Cerner Ignite).

Role: The server issuing access tokens to the client after successfully authenticating the resource owner and obtaining authorization.

Resource Server

Who: The EHR's FHIR API.

Role: The server hosting the protected resources (clinical data), capable of accepting and responding to protected resource requests using access tokens.

2. The Identity Layer: OpenID Connect (OIDC)

OAuth 2.0 was designed exclusively for authorization (granting access to APIs), not authentication (proving exactly who the user is). Because platforms needed to know user identity, they started writing custom, non-standard hacks on top of OAuth. This led to a fragmented ecosystem.

OpenID Connect (OIDC) is a standardized identity layer built directly on top of OAuth 2.0. By adding OIDC to the SMART on FHIR flow, apps can securely authenticate the user and authorize data access in a single unified step, receiving two distinct types of tokens.

Token Type Protocol Primary Purpose Analogy Format
Access Token OAuth 2.0 Used in the Authorization: Bearer header to fetch FHIR data from the Resource Server. Instructs the API what the app is allowed to do. A Valet Key. It lets the valet drive your car, but doesn't tell them who you are or let them open the glovebox. Often an Opaque String (meaningless to the client app).
ID Token OpenID Connect Provides cryptographic proof to the Client application of exactly who just logged in, when they logged in, and how. A Driver's License. It unequivocally proves your identity to the bouncer at the door. Always a JSON Web Token (JWT).

3. Unpacking the ID Token: JSON Web Tokens (JWT)

OIDC strictly specifies that the ID Token must be formatted as a JSON Web Token (JWT). A JWT is a compact, URL-safe means of representing claims to be transferred between two parties. It consists of three base64-url encoded parts separated by periods (.).

1. The Header (JOSE Header)

Tells the receiving server how to verify this token — which algorithm was used to sign it, and which specific public key to look up from the JWKS endpoint.

JWT Header

  "kid":      "jwt",
  "alg":      "RS256"
}
x5t#S256 SHA-256 thumbprint of the signing certificate. Used to locate the exact key.
kid Key ID. Points to the matching public key in the EHR's JWKS endpoint.
alg Signing algorithm. RS256 = RSA with SHA-256 (asymmetric). Never HS256 in production SMART.

3. The Signature

To create the signature part, you take the encoded header, the encoded payload, a secret, the algorithm specified in the header, and sign that.

For example, if using HMAC SHA256:

HMAC-SHA256 Signature Formula
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret
)

This guarantees the token hasn't been altered by a Man-in-the-Middle.

4. Deep Dive: SMART on FHIR Payload Claims

When an EHR issues an ID token during a SMART launch, the payload contains specific standard OIDC claims alongside a very critical SMART-specific claim.

Example Decoded SMART ID Token Payload
{
  "iss": "https://ehr.hospital.org/oauth2",
  "sub": "Practitioner/12345",
  "aud": "my-app-client-id",
  "iat": 1700000000,
  "exp": 1700003600,
  "fhirUser": "https://ehr.hospital.org/fhir/Practitioner/12345"
}

Security Claims

  • iss (Issuer): The URL of the EHR Authorization Server that issued the token. Must exactly match the EHR's known endpoint.
  • aud (Audience): Your app's client_id. If this doesn't match your app, the token was stolen from another app and must be rejected.

Lifecycle Claims

  • iat (Issued At): Unix timestamp of when the token was created.
  • exp (Expiration): Unix timestamp of when the token dies. Your backend must reject the token if the current time is past this value.

Identity Claims

  • sub (Subject): A unique, stable identifier for the user within the EHR (often their internal DB ID).
  • fhirUser: (SMART Specific) The absolute FHIR URL to read the user's demographic profile! Example: .../fhir/Patient/789.

🔍 4. SMART Configuration & Discovery

How applications dynamically discover an EHR's specific security endpoints without hardcoding URLs.

1. The Auto-Discovery Mechanism

Unlike standard APIs where endpoints (like /oauth/token) might be hardcoded in application logic, SMART on FHIR relies on dynamic discovery. Because every hospital might configure their EHR slightly differently, SMART requires the EHR to host a JSON configuration file at a standard location relative to the FHIR Base URL.

The Discovery URL: [FHIR_BASE]/.well-known/smart-configuration

2. Anatomy of the Discovery JSON

When an app boots up, it fetches this JSON file to learn exactly what capabilities the EHR server supports, where security endpoints live, and what authentication methods are required.

{

}

The Core Endpoints

How do we get these URLs? Your app must fetch them dynamically at runtime from the /.well-known/smart-configuration discovery URL.

  • authorization_endpoint: Where you redirect the user's browser for login and consent (Starts the flow).
  • token_endpoint: The backend API where your server exchanges the Auth Code for actual Access/ID tokens.

JWT Validation (JWKS)

How do you verify the token you receive isn't forged?

  • 1. Fetch Keys: Make a GET request to the jwks_uri provided in this config. It returns public keys.
  • 2. Match kid: Look at the kid in your JWT header, and find the matching key in the JWKS response.
  • 3. Verify: Use that public key alongside the `RS256` algorithm to verify the JWT signature mathematically.

Security Parameters

  • token_endpoint_auth_methods: Tells you how to authenticate. Epic allows private_key_jwt (asymmetric) for backend apps.
  • code_challenge_methods: Indicates support for S256 (PKCE) which secures mobile/SPA apps.
  • issuer: Must exactly match the iss claim inside the returned JWT!

3. Client Registration Paradigms

Before any app can utilize the discovered endpoints, the EHR must know it exists. The EHR issues a unique client_id during Registration.

Registration Type Process Common Use Case
Static Registration Developer manually logs into the EHR Vendor's Developer Portal (e.g., Epic App Orchard, Cerner Code), fills out a form, uploads JWKS URLs, and is issued a static client_id. 99% of production healthcare applications.
Dynamic Registration The app programmatically sends an HTTP POST request to the EHR's registration endpoint with its metadata and is instantly issued a client_id over the wire. Automated test suites, sandbox environments. Rarely supported in live production EHRs.

🚀 6. SMART Launch — Overview

Before choosing a launch pattern, you need to answer two questions: Is there a human in the loop? and Can your app keep a secret? The answers determine everything.

1. The Two Integration Approaches

SMART on FHIR supports two fundamentally different integration architectures. The choice is not a preference — it is determined by whether a human clinician or patient is present in the flow, or whether it is purely machine-to-machine.

User-Facing (Frontend) Integration

A human is present — a clinician inside the EHR, or a patient on their phone. The OAuth flow requires the user to authenticate and grant consent. The app receives tokens scoped to that specific person's session and data.

  • Patient apps, clinician dashboards, iFrame widgets
  • Consent screen shown to user
  • Access is scoped to one patient or one user's view
Launch patterns: Standalone Launch · Embed (EHR) Launch

System-to-System (Backend) Integration

No human is present. A server daemon, cron job, or ETL pipeline needs to access EHR data autonomously — overnight syncs, analytics exports, or automated reporting. The application itself authenticates using cryptographic keys, with no consent screen.

  • Bulk data pipelines, reporting daemons, ETL jobs
  • No user login — no consent screen
  • Uses system/ scopes, not patient/ or user/
Launch pattern: Backend Services (JWT Bearer)

2. Public vs. Confidential Clients (Frontend Apps)

Within user-facing integrations, there is a further split based on a single question: can your app's execution environment protect a secret? This determines your client type and directly dictates which security mechanism you must use.

Public Client

Cannot protect secrets. The app runs on the end-user's device — in a browser as a JavaScript SPA, or as a native mobile app. Anyone with the binary or the browser devtools can inspect it. Embedding a client_secret in this code is the equivalent of taping your house key to the front door.

Examples:

  • React / Vue / Angular SPAs in the browser
  • iOS / Android native apps
  • Desktop Electron apps
Security mechanism: PKCE — a dynamic per-session proof that replaces the static secret.

Confidential Client

Can protect secrets. The app runs on a server you control. User browsers interact with your frontend, but the OAuth token exchange happens entirely on your backend server — the secret never touches the browser. An attacker cannot read server memory remotely.

Examples:

  • Node.js / Python / Java server-side apps
  • Traditional web apps with a backend (BFF pattern)
  • Apps registered with asymmetric JWKS keys
Security mechanism: client_secret or private_key_jwt (asymmetric — preferred).

The #1 SMART security mistake: Treating a server-rendered app as a public client and skipping the secret, or embedding a client_secret in a React SPA bundle and thinking it is safe. The client type is determined by execution environment — not by what you prefer. If any part of the secret can reach a user's browser, it is a public client.

3. Choosing the Right Pattern

Use this decision tree when starting a new SMART integration:

Q1: Is there a human user authenticating?

No → Use Backend Services Launch. JWT Bearer, system/ scopes, no consent screen. → See Backend Launch section.
Yes → Continue to Q2.

Q2: Is the user already inside the EHR when they launch your app?

Yes → Use Embed (EHR) Launch. The EHR injects context (patient + encounter) automatically. SSO — no extra login. → See Embed Launch section.
No → Continue to Q3.

Q3: Does your app run entirely in the browser or on a mobile device?

Yes → Standalone Launch, Public Client. Use PKCE. No secret. User picks their context. → See Standalone Launch section.
No → Standalone Launch, Confidential Client. Token exchange on your server with private_key_jwt or client_secret.

📱 7. Standalone Launch

The user initiates the app outside the EHR — from a phone home screen, a browser bookmark, or a patient portal link. The app starts cold with no context and must acquire it via OAuth.

1. The Use Case

A patient downloads a diet tracker on their iPhone and taps "Connect to My Hospital". The app has no idea who the patient is or which FHIR server to talk to. It must: discover the EHR endpoints, redirect the user to authenticate, acquire a patient context, and exchange a short-lived code for tokens — all without ever holding a hardcoded secret.

2. Sequence Diagram

3. Step-by-Step Implementation

Step 1 — Discover EHR Endpoints

Never hardcode the authorization or token URL. Fetch them dynamically from the well-known configuration endpoint relative to the FHIR base URL the user has configured.

Step 1 · Discovery
const FHIR_BASE = 'https://fhir.ecwcloud.com/fhir/r4/FFBJCD';

const config = await fetch(
  `${FHIR_BASE}/.well-known/smart-configuration`,
  { headers: { Accept: 'application/json' } }
).then(r => r.json());

const { authorization_endpoint, token_endpoint } = config;

Step 2 — Generate PKCE Verifier & Challenge

The verifier is a cryptographically random string (43–128 unreserved ASCII characters). The challenge is its SHA-256 hash, Base64url-encoded. The verifier is stored locally; only the challenge travels over the network.

Step 2 · PKCE Generation
function generateVerifier() {
  const array = new Uint8Array(64);
  crypto.getRandomValues(array);
  return btoa(String.fromCharCode(...array))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');
}

async function generateChallenge(verifier) {
  const data = new TextEncoder().encode(verifier);
  const hash = await crypto.subtle.digest('SHA-256', data);
  return btoa(String.fromCharCode(...new Uint8Array(hash)))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');
}

const code_verifier  = generateVerifier();
const code_challenge = await generateChallenge(code_verifier);
const state          = crypto.randomUUID(); // ≥122 bits entropy

// Persist until the redirect returns
sessionStorage.setItem('pkce_verifier', code_verifier);
sessionStorage.setItem('oauth_state',   state);

Step 3 — Build & Redirect to Authorization URL

Construct the URL with all required parameters and redirect the browser. The aud parameter must be the FHIR server base URL — this binds the token to a specific server and prevents token reuse attacks.

Step 3 · Authorization Redirect
const params = new URLSearchParams({
  response_type:         'code',
  client_id:             'YOUR_CLIENT_ID',
  redirect_uri:          'https://your-app.com/callback',
  scope:                 'launch/patient patient/Patient.rs patient/Observation.rs openid fhirUser offline_access',
  state,
  aud:                   FHIR_BASE,
  code_challenge,
  code_challenge_method: 'S256',
});

window.location.assign(`${authorization_endpoint}?${params}`);

Step 4 — Handle the Callback & Validate State

The EHR redirects back to your redirect_uri with ?code=…&state=…. Always validate the state before proceeding — a mismatch indicates a CSRF attack.

Step 4 · Callback Validation
const urlParams    = new URLSearchParams(window.location.search);
const code         = urlParams.get('code');
const returnedState = urlParams.get('state');

if (returnedState !== sessionStorage.getItem('oauth_state')) {
  throw new Error('State mismatch — possible CSRF attack. Aborting.');
}

const code_verifier = sessionStorage.getItem('pkce_verifier');
sessionStorage.removeItem('pkce_verifier');
sessionStorage.removeItem('oauth_state');

Step 5 — Exchange Code for Tokens

POST the authorization code along with the original code_verifier (not the challenge) to the token endpoint. The EHR re-hashes the verifier and confirms it matches the challenge sent in Step 3.

Step 5 · Token Exchange
const tokenResponse = await fetch(token_endpoint, {
  method: 'POST',
  headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
  body: new URLSearchParams({
    grant_type:    'authorization_code',
    code,
    redirect_uri:  'https://your-app.com/callback',
    code_verifier, // ← original verifier, NOT the challenge
    client_id:     'YOUR_CLIENT_ID',
  }),
}).then(r => r.json());

// tokenResponse contains:
// { access_token, token_type, expires_in, scope,
//   patient, id_token, refresh_token? }

Step 6 — Call the FHIR API

Use the access_token as a Bearer token. The patient field in the token response is the FHIR Patient resource ID the user selected during login.

Step 6 · Fetch FHIR Data
const { access_token, patient } = tokenResponse;

const patientData = await fetch(
  `${FHIR_BASE}/Patient/${patient}`,
  { headers: { Authorization: `Bearer ${access_token}` } }
).then(r => r.json());

const observations = await fetch(
  `${FHIR_BASE}/Observation?patient=${patient}&category=laboratory`,
  { headers: { Authorization: `Bearer ${access_token}` } }
).then(r => r.json());
Never store tokens in localStorage. Use sessionStorage or an in-memory store. localStorage is accessible to any script on your page — a single XSS vulnerability exposes every token you've stored.

🏥 8. Embed Launch (EHR Launch)

The EHR opens your app from within its own interface. The clinician never logs in twice — the active patient and encounter are injected automatically into your token response.

1. The Use Case

Scenario: Dr. House is viewing Patient John Doe's chart in ECW. He clicks "Open MAIA" in the sidebar. Your app appears in an iFrame — already scoped to John Doe, with the active encounter ID ready to use. No login. No patient search. No manual context setup.

The mechanism is the launch parameter — an opaque, short-lived token the EHR generates at the exact moment it navigates to your launch URL. It encodes the clinician's active session context. Your app echoes it back during the authorization request, and the EHR's auth server resolves it into a context-aware access token.

launch — Opaque Context Token

What it is: A short-lived, opaque string the EHR generates at the moment it opens your app. It is a sealed envelope that contains the EHR's knowledge of the active session — which patient chart is open, which encounter is active, which clinician is logged in. Your app cannot read its contents.

What you do with it:

  • Store it immediately in sessionStorage or your server session.
  • Echo it back as the launch query parameter in your authorization request.
  • Include launch in your scope string to signal you're doing an EHR launch.
  • Never attempt to decode, store long-term, or reuse it. It is valid only for a single authorization request and expires in seconds.

⚙️ 9. Backend Launch (System-to-System)

No user. No browser. No consent screen. A server daemon authenticates directly to the EHR using asymmetric cryptography — signing a JWT with its private key as proof of identity.

1. The Problem: No Human "Consent"

For background tasks — overnight appointment syncs, quality measure calculations, bulk data exports — there is no human to click "I Agree". The application itself must prove its identity. Sharing a client_secret is too weak for this trust level; instead, the EHR and your server establish trust through Public Key Cryptography registered at onboarding time.

3. The Token Exchange Anatomy — What Your Server Sends & Gets

Unlike Standalone or Embed launch, there is no browser redirect, no authorization code, and no user interaction. Your server makes a single POST to the token endpoint with a signed JWT as its identity proof. Here is the exact anatomy of that exchange.

Part A — The JWT Assertion Claims

The JWT you sign is your server's identity credential. It must include exactly five claims — no more, no less. Missing any one will cause the EHR to reject the request.

iss & sub — Who You Are

Both are set to your registered client_id. This tells the EHR which application is making the request. The EHR uses iss to look up your registered JWKS URL and fetch your public key.

Example
{
  "iss": "mindbowser-maia-backend",
  "sub": "mindbowser-maia-backend"
}

aud — Where You're Sending It

The exact URL of the token endpoint you're posting to. This binds the JWT to a specific server — a JWT signed for Epic's token endpoint cannot be replayed against ECW's token endpoint.

Example
{
  "aud": "https://fhir.ecwcloud.com/oauth2/token"
}

exp — Short-Lived Window

Unix timestamp — must be within 5 minutes of the current time. The EHR will reject assertions with an expiry further than 300 seconds in the future. This forces key rotation and prevents stale credentials from being reused.

Example (now + 300s)
exp: Math.floor(Date.now() / 1000) + 300

jti — Replay Prevention

A unique identifier per request — typically a UUID v4. The EHR caches seen jti values for the JWT's lifetime window. Sending the same jti twice (even with a fresh exp) results in a invalid_grant rejection.

Example
jti: crypto.randomUUID() // fresh every call

Part B — The Full POST /token Request

Backend Token Request — form-encoded body
POST https://fhir.ecwcloud.com/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type            = client_credentials
client_assertion_type = urn:ietf:params:oauth:client-assertion-type:jwt-bearer
client_assertion      = eyJhbGciOiJSUzM4NCI...  ← your signed JWT
scope                 = system/Patient.rs system/Observation.rs system/Appointment.rs

Part C — The Token Response

The backend token response is deliberately minimal. There is no patient, no encounter, no identity token — the system scope has no user context to attach.

Backend Token Response (ECW / Epic)
{
  "access_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6Imp3dCJ9...",
  "token_type":   "bearer",
  "expires_in":   300,
  "scope":        "system/Patient.rs system/Observation.rs system/Appointment.rs"
}

access_token

A signed JWT. Use it as Authorization: Bearer <token> on every FHIR API call. Valid for expires_in seconds — cache it and reuse it until near expiry.

expires_in

Usually 300 seconds (5 min) per the SMART Backend Services spec. Never hardcode this — read the value from the response and subtract a 30-second buffer before requesting a fresh token.

scope

The EHR may grant a narrower scope than requested. Always parse the returned scope string and verify you received the permissions your job needs before executing bulk queries.

No patient context — by design. Backend tokens carry system/ scopes that grant access to all patients on the server (subject to EHR admin approval). You supply the patient ID yourself in every FHIR query: GET /Patient/abc123. There is no EHR-injected context because there is no active user session.

4. The Cryptographic Trust Model

1. Generate a Keypair

Generate an RSA (2048+ bit) or EC keypair. The private key never leaves your server. Store it in a secrets manager (Vault, AWS Secrets Manager, env variable). The corresponding public key is what the EHR uses to verify your identity.

2. Host a JWKS Endpoint

Convert your public key to JWK format and expose it at a publicly reachable URL (e.g., https://your-app.com/.well-known/jwks.json). This URL is registered with the EHR during onboarding. The EHR fetches it on-demand to verify your JWT signatures.

3. Sign a JWT Assertion

Instead of a password, you construct a short-lived JWT, sign it with your private key, and send it to the token endpoint. The EHR fetches your public key, verifies the signature, and issues an access token. No shared secret ever travels over the network.

🛡️ 5. SMART Scopes & Permissions

Every scope is a contract — it specifies exactly whose data, which resource, and what operations your app is permitted to perform.

1. v1 vs v2 — A Quick History

SMART scopes have gone through two major generations. Most EHRs today support both, but v2 is the current standard and what all new apps should target.

SMART v1 2016 – 2022

Scopes used a simple .read / .write / .* suffix. Coarse-grained — no way to say "read-only, no delete". A .read scope granted both read and search; you couldn't separate them.

v1 Scope Examples
patient/Observation.read    → read + search only
patient/Condition.write     → create + update + delete
patient/*.* OR patient/*.*  → everything (very broad)

SMART v2 2022 – present

Replaced the blunt read/write with individual CRUD letters c · r · u · d · s. Each operation can now be granted independently — an app can search without being able to delete. Adds optional fine-grained query filters and system-level scopes.

v2 Scope Examples
patient/Observation.rs      → read + search, no write
patient/Condition.cru       → create, read, update only
system/*.rs                 → bulk read, server-wide
EHR backward compatibility: Servers advertising the permission-v1 capability automatically map legacy scopes — .read → .rs, .write → .cud, .* → .cruds. All examples from here on use v2 syntax.

2. The v2 Scope Formula

SMART v2 scopes are machine-parseable strings built from three parts. Think of them as answering three questions in sequence: Who owns it? → What resource? → What can I do?

SMART v2 Scope Syntax
[context] / [ResourceType | *] . [c][r][u][d][s]  [? param=value ]
   ↑               ↑                     ↑                    ↑
patient|user    Observation           in-order             optional
 system           Condition           subset of           fine-grained
                    *                 "cruds"              filter

Context

The principal — who's data boundary applies?

  • patient — one specific patient's chart
  • user — all records the logged-in user can access
  • system — server-to-server, no human involved

Resource Type

The FHIR resource or a wildcard.

  • Observation, Condition, MedicationRequest…
  • * — all resource types (use sparingly)
  • EHR may grant a narrower scope than requested.

CRUD Actions

Letters must appear in order: c · r · u · d · s

  • c create   r read   u update
  • d delete   s search
  • ✅ .rs   ✅ .cruds   ❌ .sr

3. patient/ — Patient-Level Scopes

Restricts access to a single patient's data. The patient is established either by the EHR Launch context (Embed Launch) or by requesting launch/patient in Standalone Launch. This is the most common context for clinical and patient-facing apps.

If an app requests patient/*.rs and the EHR grants it, the EHR must establish a patient in context. An EHR may refuse patient/ scopes if no valid launch or launch/patient context exists.

patient/Patient.rs Read & search the patient's demographic record Any app that displays who the patient is
patient/Observation.rs Read labs, vitals, smoking status, BMI Clinical dashboard, diet & activity tracker
patient/Condition.rs Read problem list and diagnoses Care plan, chronic disease management
patient/MedicationRequest.rs Read active and historical prescriptions Medication reconciliation, pharmacy integration
patient/AllergyIntolerance.rs Read allergy and intolerance list Clinical safety checks, pharmacy apps

4. user/ — User-Level Scopes

Grants access to all data the currently logged-in clinician is permitted to see, spanning multiple patients. The data boundary is the user's system permissions, not a single chart. If a clinician is only authorized to see their own patients, the scope is still enforced by the EHR's underlying RBAC — SMART scopes cannot exceed those permissions.

Scope What it Unlocks Typical Use Case
user/Patient.rs Search and read patients the user is authorized to access Patient list views, patient selection UI
user/Appointment.cruds Full CRUD over appointments in the user's schedule Scheduling apps, calendar integration
user/Observation.rs Read observations across all of the user's patients Population health monitoring dashboards
user/Condition.rs Read diagnoses across all accessible patients Panel management, care gap identification
user/Practitioner.rs Read provider directory information Care team lookup, referral coordination
user/*.rs Read all resources the user can access across their panel Broad clinical decision support, audit tools

5. system/ — System-Level Scopes

Used exclusively in Backend (machine-to-machine) flows with no human user involved. The client application itself is the principal. These require explicit trust established during app registration — the EHR's administrator must approve system-level access. Never use system/ scopes in user-facing apps.

Scope What it Unlocks Typical Use Case
system/Patient.rs Read all patient records on the server Patient index sync, deduplication pipelines
system/Observation.rs Read all observations across the entire EHR Quality measure calculation, BI export
system/Appointment.rs Read all appointments facility-wide Scheduling analytics, no-show prediction
system/*.rs Read-only bulk access to all resources FHIR Bulk Data ($export) for analytics or data warehouses
system/*.cruds Full read/write access system-wide EHR migration tools, master data management (high risk)

6. Launch Context Scopes

These scopes don't control resource access — they control what contextual information the EHR attaches to the token response. Without them, your app won't know which patient or encounter is active.

launch

Used in Embed (EHR) Launch only. Tells the auth server to associate the authorization request with the current EHR session. Your app passes back the launch=abc123 token the EHR injected into your URL. The EHR then automatically attaches the active patient and encounter to your token response.

launch/patient

Used in Standalone Launch. Tells the EHR: "I need a patient in context — show a patient selector UI during login." After the user picks a patient, the EHR returns a patient ID in the token response. Without this scope, patient/ resource scopes have no patient to apply to.

launch/encounter

Requests that an encounter be established in context (with an encounter selector if needed). The token response will include an encounter ID. Use this when your app needs to know the specific visit — not just the patient — for documentation or billing workflows.

Token Response — Launch Context Fields
{
  "access_token": "eyJhbGci...",
  "token_type":   "Bearer",
  "expires_in":   3600,
  "scope":        "launch/patient patient/Observation.rs openid fhirUser",
  "patient":      "12b3c4d5",       // ← returned because launch/patient was requested
  "encounter":    "987654321",      // ← returned because launch/encounter was requested
  "need_patient_banner": true,      // ← UX hint: should your app show a patient header?
  "smart_style_url": "https://..."  // ← EHR theme colors for your UI to match
}

7. Identity & Session Scopes

These are OpenID Connect scopes that unlock user identity information and session management. They are independent of clinical data access and work alongside any resource scopes.

openid

Required for OIDC. Instructs the authorization server to issue an id_token alongside the access token. Without this scope, you receive no ID token and cannot verify the identity of who just logged in. Always pair with fhirUser.

fhirUser

SMART-specific OIDC extension. Adds the fhirUser claim to the ID token — an absolute FHIR URL pointing to the logged-in user's FHIR resource (e.g., Practitioner/12345 or Patient/789). Fetch that URL to get their full demographics. This is what your app uses to display "Logged in as Dr. Smith."

profile

Standard OIDC scope. Requests basic user profile claims in the ID token: name, given_name, family_name, locale, etc. Less useful than fhirUser in healthcare contexts since FHIR resources contain richer structured data.

offline_access

Requests a Refresh Token valid indefinitely (per server policy). Use for background sync daemons or apps that need to access data after the user closes the browser. The EHR may require additional user consent for this scope.

online_access

Requests a Refresh Token valid only while the end-user remains online / session-active. When the user logs out of the EHR, the refresh token is automatically invalidated. Use for session-bound apps like embedded iFrames.

Minimal scope principle: Always request the smallest set of scopes that satisfies your use case. EHRs will grant a subset of what you request — your app must handle receiving fewer permissions gracefully. Requesting patient/*.cruds as a catchall is a red flag during App Orchard review.

8. Fine-Grained Scope Filters (v2)

SMART v2 allows appending FHIR search parameters directly to a scope, creating a read access filter that is enforced at the authorization level — not just at query time.

Fine-Grained Scope — Lab Results Only
patient/Observation.rs?category=http://terminology.hl7.org/CodeSystem/observation-category|laboratory

This tells the EHR: "Grant read/search on Observations, but only for the laboratory category." The server must enforce this filter even if the client queries without the parameter. This is a powerful tool for privacy-by-design — a mental health app could request patient/Observation.rs?category=...survey and be structurally prevented from ever seeing lab data.

Healthconnect Integration Module

The ConnectHealth ECW Integration Submodule wraps all eClinicalWorks OAuth and FHIR behavior behind a clean, vendor-agnostic TypeScript surface — handling PKCE, JWKS, token rotation, and audit logging so you never touch raw OAuth mechanics.

Factory classes not yet implemented. The README documents OAuthServiceFactory and FHIRServiceFactory as the intended entry points — but these factory classes do not exist yet in the codebase. Only the underlying services are implemented: ECWOAuthService, ECWFHIRService, and their shared ECWIntegrationService facade. The factory wrapper needs to be built to store config-per-integration and expose a clean call site.

1. What the Submodule Abstracts

Mandatory PKCE (S256)

Every authorization flow automatically generates a code_verifier and code_challenge. You never call the raw OAuth endpoints — PKCE is always enforced.

JWT Bearer (RS384)

Backend tokens are built, signed with your RSA-384 private key, and sent to ECW's token endpoint automatically. The jti, exp, and aud claims are constructed internally.

JWKS & Discovery

Public-key hosting and SMART endpoint auto-discovery (.well-known/smart-configuration) are handled internally. Your code only passes a fhir_base_url.

90-Day Refresh Tokens

ECW issues a new refresh_token on every refresh call — the previous one is immediately invalidated. The submodule surfaces this clearly in TokenResponse so you always store the latest.

Full FHIR R4 Surface

Read, search, create, update, delete, batch, transaction, bulk export, and capability statement — all as typed async methods on ECWFHIRService.

Audit Logging

Every OAuth and FHIR operation is automatically audit-logged by the wrapper services. Your application code needs no logging boilerplate.

3. The ECWClient Class — Bound-Config Pattern

Wrap ECWIntegrationService in a class that captures the config once in its constructor. Call sites never pass config again — just call the method.

ECWClient.ts
import Container from 'typedi';
import { ECWIntegrationService } from
  'healthconnect-ecw-integration-submodule-node-express';
import type { ECWConfig, TokenResponse, FHIRResourceResponse, Bundle }
  from 'healthconnect-ecw-integration-submodule-node-express';

/**
 * Binds an ECW config to ECWIntegrationService.
 * Instantiate once per tenant; reuse across the application.
 */
export class ECWClient {
  private readonly svc: ECWIntegrationService;

  constructor(private readonly config: ECWConfig) {
    this.svc = Container.get(ECWIntegrationService);
  }

  // ── Auth ────────────────────────────────────────────────────────────
  getBackendToken(): Promise<TokenResponse> {
    return this.svc.generateBackendToken(this.config);
  }

  generateAuthUrl(state: string, redirectUri: string) {
    return this.svc.generateStandaloneAuthUrl(this.config, state, redirectUri);
  }

  exchangeCode(code: string, verifier: string, redirect: string, iss?: string) {
    return this.svc.exchangeCodeForToken(this.config, code, verifier, redirect, iss);
  }

  handleEmbedLaunch(iss: string, launch: string, redirect: string, state?: string) {
    return this.svc.handleEmbedLaunch(this.config, iss, launch, redirect, state);
  }

  refreshToken(token: string, patientId?: string): Promise<TokenResponse> {
    return this.svc.refreshToken(this.config, token, patientId);
  }

  introspectToken(token: string) {
    return this.svc.introspectToken(this.config, token);
  }

  // ── FHIR ────────────────────────────────────────────────────────────
  getResource(type: string, id: string, token: string): Promise<FHIRResourceResponse> {
    return this.svc.getResource(this.config, type, id, token);
  }

  searchResources(type: string, params: Record<string, string>, token: string): Promise<Bundle> {
    return this.svc.searchResources(this.config, type, params, token);
  }

  createResource(type: string, resource: object, token: string): Promise<FHIRResourceResponse> {
    return this.svc.createResource(this.config, type, resource, token);
  }

  updateResource(type: string, id: string, resource: object, token: string): Promise<FHIRResourceResponse> {
    return this.svc.updateResource(this.config, type, id, resource, token);
  }

  deleteResource(type: string, id: string, token: string): Promise<void> {
    return this.svc.deleteResource(this.config, type, id, token);
  }

  batch(bundle: object, token: string): Promise<Bundle> {
    return this.svc.batch(this.config, bundle, token);
  }

  transaction(bundle: object, token: string): Promise<Bundle> {
    return this.svc.transaction(this.config, bundle, token);
  }
}

4. OAuth Flows — Full Reference

4a — Backend Token (System-to-System)

No user. No browser. Pass your credentials config and get an access token scoped with system/ permissions.

Backend Token — Config + Call
import { ecw } from './ecwSingleton'; // The pre-configured ECWClient

const tokens = await ecw.getBackendToken();

console.log(tokens.access_token); // Bearer token
console.log(tokens.expires_in);   // 3600s — cache and reuse
console.log(tokens.token_type);   // "Bearer"
Token caching: Backend tokens expire in 3600 seconds. Cache access_token and subtract a 30-second buffer before requesting a fresh one. Re-calling generateBackendToken on every FHIR request is wasteful and will hit ECW rate limits.

4b — Standalone Launch (Patient / Provider-Facing)

Three-step flow: generate auth URL → user authorizes → exchange code for tokens. PKCE is automatic.

Step 1 · Generate Authorization URL
import { ecw } from './ecwSingleton';

const { authUrl, code_verifier } = await ecw.generateAuthUrl(
  'csrf-state-token',           // random state for CSRF protection
  'https://yourapp.com/callback'
);

// IMPORTANT: Persist code_verifier (sessionStorage / server session)
// before redirecting — you need it in Step 2.
// Redirect the user to authUrl.
Step 2 · Exchange Code for Tokens (in your /callback handler)
// code_verifier retrieved from where you stored it in Step 1
const tokens = await ecw.exchangeCode(
  req.query.code,               // authorization_code from ECW redirect
  code_verifier,                // PKCE verifier from Step 1
  'https://yourapp.com/callback'
);

console.log(tokens.access_token);  // use for FHIR calls
console.log(tokens.refresh_token); // valid for 90 days — store securely
console.log(tokens.patient);       // Patient ID in context (if launch/patient requested)
Step 3 · Refresh When Token Expires
// CRITICAL: A NEW refresh_token is issued — the old one becomes invalid.
// Always replace your stored refresh_token with newTokens.refresh_token.
const newTokens = await ecw.refreshToken(
  tokens.refresh_token,
  tokens.patient               // required if original token had patient context
);

// Store newTokens.refresh_token — it replaces the previous one immediately.

4c — Embed (EHR) Launch

ECW navigates your /launch URL with ?iss=…&launch=…. Pass both values directly into handleEmbedLaunch — the submodule handles PKCE generation, SMART endpoint discovery, and authorization URL construction.

Step 1 · Handle /launch — Build Redirect URL
import { ecw } from './ecwSingleton';

// iss and launch come directly from ECW's query parameters
const { redirect, code_verifier } = await ecw.handleEmbedLaunch(
  req.query.iss,               // 'https://fhir.eclinicalworks.com/fhir/r4'
  req.query.launch,            // opaque launch token from ECW
  'https://yourapp.com/embed-callback',
  'state-value'
);

// Store code_verifier, then redirect the browser to redirect URL.
res.redirect(redirect);
Step 2 · Handle /embed-callback — Exchange Code
const tokens = await ecw.exchangeCode(
  req.query.code,
  code_verifier,                                     // from Step 1
  'https://yourapp.com/embed-callback',
  'https://fhir.eclinicalworks.com/fhir/r4'         // iss for SMART discovery
);

console.log(tokens.patient);    // Active patient — injected by EHR
console.log(tokens.encounter);  // Active encounter — injected by EHR

Batch & Transaction Bundles

Batch

Each entry in the bundle is executed independently. One failure does not block the others. Each entry in the response bundle has its own status code.

const result = await ecwFHIR.batch(config, {
  resourceType: 'Bundle',
  type: 'batch',
  entry: [
    { request: { method: 'GET', url: 'Patient/123' } },
    { request: { method: 'GET',
        url: 'Observation?patient=123' } }
  ]
}, accessToken);
// result.entry[0].response.status: "200 OK"
// result.entry[1].response.status: "200 OK"

Transaction

All-or-nothing. If any entry fails, the entire bundle is rolled back. Use for multi-resource writes that must be atomic (e.g., Patient + Observation together).

const result = await ecwFHIR.transaction(config, {
  resourceType: 'Bundle',
  type: 'transaction',
  entry: [
    { resource: { resourceType: 'Patient',
        name: [{ family: 'Smith' }] },
      request: { method: 'POST', url: 'Patient' } },
    { resource: { resourceType: 'Observation',
        status: 'final', /* ... */ },
      request: { method: 'POST',
        url: 'Observation' } }
  ]
}, accessToken);
// All entries committed, or all rolled back

8. Error Handling

Both services throw a CustomError with a status code and descriptive message on any failure. Wrap all calls in try/catch.

Error Handling Pattern
try {
  const patient = await ecwFHIR.readResource(config, 'Patient', 'bad-id', token);
} catch (error) {
  // error.status:  404
  // error.message: "Resource not found: Patient/bad-id"

  if (error.status === 401) {
    // Token expired — refresh and retry
    const newTokens = await ecwOAuth.refreshToken(config, storedRefreshToken);
  }
}
Status Meaning Common Cause
400 Bad Request Invalid search params, missing PKCE, malformed resource
401 Unauthorized Access token expired or revoked — call refreshToken
403 Forbidden Scope not granted — check your auth_scope config
404 Not Found Resource ID doesn't exist on this ECW instance
422 Unprocessable FHIR resource failed validation — check required fields
500 Server Error ECW API or submodule internal failure
Security checklist: Never commit private_key or client_secret to git. Load them from environment variables or a secrets manager (AWS Secrets Manager, Vault). Token exchange must always happen on the server — the submodule's /workflow/auth/{integrationId}/callback handles this. Always store the latest refresh_token — the previous one is invalidated the moment ECW issues a new one.
